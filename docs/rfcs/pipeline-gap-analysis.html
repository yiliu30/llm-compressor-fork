Layout Difference (Important)

compressed-tensors layout: Packs 32 // num_bits values into one int32 (LSB-first, with wasted bits for non-divisors of 32).
humming layout: Packs 32 values into num_bits int32 words as a contiguous bitstream (no wasted bits).

This is already handled by humming's convert_humming() which unpacks from compressed-tensors format and repacks into MMA-optimized layout.

Recommended Work Plan

Tier 1 compressed-tensors (Required)

  1. Relax can_compress in compressors/pack_quantized/base.py to accept num_bits in range(1, 9)
  2. Add generic WNA16 preset schemes: W3A16, W5A16, W6A16, W7A16 (symmetric INT, group-128)
  3. Add unit tests for pack/unpack round-trip at 3-bit and 5-bit

Tier 2 vLLM Integration (Required)

  1. Verify HummingConfig + convert_humming() handles 3/5-bit packed weights from compressed-tensors format
  2. If not working: fix the conversion in humming's compressed_tensors.py schema loader
  3. If working: no vLLM changes needed (user uses --quantization humming)

Tier 3 End-to-End Validation (Required)

  1. Compress TinyLlama at 3-bit GPTQ via llm-compressor
  2. Load in vLLM with --quantization humming
  3. Run inference, compare perplexity against FP16 baseline

Key Decision Point

Q: Should sub-bit work via --quantization compressed-tensors (default path)?

If no (humming-only): Scope is minimal — just relax the guard + add presets in compressed-tensors. Users pass --quantization humming.

If yes: Requires extending vLLM's WNA16 scheme to route sub-bit widths to humming kernels (instead of Marlin). More invasive vLLM changes.

Recommendation: Start with humming-only path. The default compressed-tensors path can be added later as a follow-up.
