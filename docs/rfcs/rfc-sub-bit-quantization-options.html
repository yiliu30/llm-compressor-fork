RFC: Sub-Bit Quantization Support (3/5/6/7-bit)

Author: Yi Liu  |  Date: 2026-05-20  |  Status: Draft

1. Summary

This RFC compares two ways to enable 3/5/6/7-bit weight quantization through llm-compressor → compressed-tensors → vLLM → humming. The core question is where to resolve the layout mismatch for odd bit-widths:

  1. Option A: keep the current compressed-tensors packed layout and teach humming to load it correctly.
  2. Option B: add a humming-aligned dense packing path in compressed-tensors for odd bit-widths and keep humming/vLLM largely unchanged.

Recommendation: choose Option B. It keeps the runtime path simpler, avoids adding new sub-bit logic in humming/vLLM, and only changes newly generated odd-bit checkpoints.

2. Scope

  • In scope: weight-only INT 3/5/6/7-bit checkpoints produced by llm-compressor / compressed-tensors and loaded in vLLM through the existing --quantization humming path.
  • In scope: high-level repo changes and option comparison.
  • Out of scope: a second vLLM rollout stage, generic auto-routing through vLLM's mixed-precision kernel framework, and new default kernel selection logic.
  • Out of scope: 1-bit / 2-bit support in this RFC. 4-bit and 8-bit already work and remain unchanged.

3. Current Mismatch

For 4-bit and 8-bit, compressed-tensors and humming are effectively layout-compatible. For odd bit-widths, they are not.

Key observation: 4-bit and 8-bit do not need special treatment here. The real design choice is how to handle 3/5/6/7-bit serialization and loading.

4. Packing Solutions

The underlying technical decision is the packing format used for odd bit-width checkpoints. The two RFC options are downstream consequences of this packing choice.

Packing solutionDescriptionImplication
Padded CT packingKeep the current compressed-tensors layout, where each int32 stores 32 // num_bits values and any remaining bits are wasted for odd widths.humming must explicitly understand and convert this padded odd-bit layout at load time.
Dense humming-aligned packingIntroduce a dense odd-bit layout for 3/5/6/7-bit weights so 32 values are stored across num_bits int32 words without wasted bits.The runtime path stays simpler because the checkpoint format is already closer to humming's expectation.
Important: this packing decision only matters for odd bit-widths. 4-bit and 8-bit can remain unchanged because they are already layout-compatible in practice.

Mapping to the RFC options:

  • Option A corresponds to keeping the padded compressed-tensors packing and adapting humming.
  • Option B corresponds to introducing dense odd-bit packing in compressed-tensors and keeping the runtime simpler.

5. The Two Options

4.1 Option A — Keep Current CT Layout, Adapt Humming

Compressed-tensors continues to emit its current padded pack-quantized format for odd bit-widths. Humming learns how to interpret that format correctly, including packed-shape inference, zero-point shape handling, and repacking to the internal MMA-friendly layout.

Implication: the complexity stays in the runtime path. Every odd-bit checkpoint load must handle the padded compressed-tensors layout correctly.

4.2 Option B — Add Dense Odd-Bit Packing in CT, Keep Runtime Simple

Compressed-tensors adds a new dense packing path for 3/5/6/7-bit weights that matches what humming already expects. 4-bit and 8-bit remain on the existing path because they are already compatible. Humming and vLLM mostly verify behavior rather than grow new conversion logic.

Implication: the complexity stays in the offline serialization path. Newly generated odd-bit checkpoints become smaller and easier for the runtime to consume.

8. Recommendation

Choose Option B.

  • It keeps sub-bit-specific logic out of the serving-time critical path.
  • It aligns newly generated odd-bit checkpoints with the format humming already wants.
  • It avoids growing a second set of odd-bit conversion rules in humming.
  • It does not disturb existing 4-bit / 8-bit models.
Tradeoff: Option B is the better engineering path only if we are comfortable defining a new serialization behavior for newly generated 3/5/6/7-bit compressed-tensors checkpoints. If preserving the current odd-bit disk format for cross-tool compatibility is mandatory, choose Option A instead.

9. Validation

  1. Compress a small model at 3-bit and 5-bit with GPTQModifier and AutoRoundModifier.
  2. Load with vllm serve --quantization humming and verify successful generation.
  3. Validate edge shapes, not just one clean hidden size: non-trivial packing boundaries, tensor-parallel partitions, and asymmetric zero-point cases if supported.
  4. Compare quality against 4-bit and verify no NaN / crash / shape mismatch.

9.2 Option A Details

  • Audit humming/schema/compressed_tensors.py beyond convert_humming(). Packed tensor shape inference and packed zero-point shapes matter too.
  • Use ceil-div style formulas where compressed-tensors stores padded odd-bit data.
  • Call out the current act-order limitation explicitly: compressed-tensors checkpoints with actorder="group" are not automatically in scope for humming today.

9.3 Option B Details

  • Add a dense odd-bit packing path in compressed-tensors for 3/5/6/7-bit weights.
  • Keep the existing 4-bit / 8-bit packing path unchanged.
  • Dense odd-bit packing should be paired with a matching unpack path for tests and offline inspection.
  • If we require K % 32 == 0 for dense packing, document that clearly and validate it early.
